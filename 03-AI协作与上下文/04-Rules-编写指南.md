# Rules 编写指南

这篇文档只回答一个问题：

**项目里的 Rules 到底怎么写，AI 才更容易稳定执行。**

---

## 1. 什么是 Rule

Rule 就是：

> 你希望 AI 在项目里稳定遵守的明确规则。

比如：

- 文档统一用中文
- commit message 使用 `type: summary`
- 不要提交 `.env`
- 改完文档后要更新 `CHANGELOG.md`

Rule 不一定非要叫 Rule，它可以写在：

- `AGENTS.md`
- `CLAUDE.md`
- `SKILL.md`
- 模板说明文档

---

## 2. 好的 Rule 长什么样

好的 Rule 通常有 5 个特点：

### 具体

不要写：

```text
写得专业一点
```

更好写法：

```text
commit message 使用 `type: summary` 格式，type 仅限 `feat`、`fix`、`docs`、`refactor`、`chore`、`test`
```

### 可执行

不要写：

```text
尽量清晰
```

更好写法：

```text
每次修改完成后，先运行链接检查，再更新 `TASKS.md` 和 `DECISIONS.md`
```

### 有边界

不要让一条 Rule 既管文档、又管代码、又管发布。

### 可验证

看完之后应该知道怎么检查是否做到。

### 稳定

长期规则才适合写进项目规则文件，临时要求不要硬塞进去。

---

## 3. 坏 Rule 和好 Rule 对比

### 例子 1

坏：

```text
注意代码质量
```

好：

```text
修改前先阅读相关文件；修改后至少做一次与改动直接相关的验证，并在最终说明中写明是否已验证
```

### 例子 2

坏：

```text
文档写完整一点
```

好：

```text
文档使用中文；新增教程时至少包含“适用场景、步骤、常见问题、继续阅读”四个部分
```

### 例子 3

坏：

```text
规范提交
```

好：

```text
commit message 使用 `type: summary`；一次提交只表达一件主要事情；push 前先查看 `git diff --staged`
```

---

## 4. Rules 应该写到哪里

### 写进 `CLAUDE.md`

适合放：

- 项目目标
- 长期有效的工作规则
- 常用命令
- 关键目录说明

### 写进 `AGENTS.md`

适合放：

- 协作协议
- 文档更新要求
- Git 工作流
- 命名规范
- 完成标准

### 写进 `SKILL.md`

适合放：

- 这个 Skill 在什么场景触发
- 这个 Skill 的操作步骤
- 这个 Skill 的输入输出要求

### 写进 `.claude/settings.json`

适合放：

- 权限限制
- 环境变量
- 工具行为配置

也就是说：

**不是所有 Rule 都该写在 Markdown 里。**

---

## 5. 一个推荐的 Rule 写法模板

你可以直接套这个结构：

```markdown
## 规则标题

- **目的**：这条规则解决什么问题
- **适用范围**：什么时候适用
- **要求**：必须做什么
- **禁止项**：不要做什么
- **验证方式**：怎么确认执行到了
```

例如：

```markdown
## Commit 规则

- **目的**：让提交历史可读、可追溯
- **适用范围**：所有 Git 提交
- **要求**：commit message 使用 `type: summary`
- **禁止项**：不要使用 `update`、`final`、`fix bug`
- **验证方式**：`git log --oneline` 可一眼看出每次提交的类型和对象
```

---

## 6. 规则不要写得太多

常见误区是：一上来写 100 条规则。

结果通常是：

- AI 不容易抓住重点
- 人也不会维护
- 很多规则互相重复

更好的做法是：

1. 先写最关键的 5 到 10 条
2. 先覆盖高频问题
3. 以后根据真实踩坑再补

---

## 7. 什么规则最值得优先写

如果你是新项目，优先写这几类：

- 语言与文档规则
- Git / commit / 分支规则
- 敏感信息规则
- 测试与验证规则
- 任务完成后要更新哪些文件
- 目录与命名规范

这几类通常比“代码风格哲学”更先产生实际价值。

---

## 8. 什么时候要改 Rule

这几种情况最值得更新规则：

- 同一个错误重复出现了两次以上
- 不同协作者对同一件事理解不一致
- 项目结构已经变了
- 原规则太模糊，导致执行结果漂移

更新规则时，建议同步更新：

- `AGENTS.md` 或 `CLAUDE.md`
- `DECISIONS.md`
- 必要时更新 `CHANGELOG.md`

---

## 9. 一个适合 starter 项目的基础 Rule 清单

可以作为最低配参考：

- 文档统一使用中文
- 相对路径必须可检查、可跳转
- 修改前先阅读相关文件
- commit message 使用 `type: summary`
- 不提交密钥、Token、`.env`
- 任务完成后更新 `TASKS.md` 和 `DECISIONS.md`
- 对外可见变更更新 `CHANGELOG.md`
