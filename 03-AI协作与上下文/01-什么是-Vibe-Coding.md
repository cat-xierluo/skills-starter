# 什么是 Vibe Coding？

Vibe Coding 可以先理解成一句话：

**你负责描述目标，AI 负责推进实现。**

它不是“完全不用脑子写代码”，而是把你的主要工作从“手敲每一行代码”，变成“定义目标、拆解问题、验收结果、持续修正”。

## 它和传统写代码有什么区别

| 对比项 | 传统 Coding | Vibe Coding |
|------|-------------|-------------|
| 主要动作 | 人手写实现细节 | 人描述目标，AI 辅助实现 |
| 核心能力 | 语法、框架、调试经验 | 表达能力、判断能力、验收能力 |
| 开发节奏 | 写一段，跑一段 | 提需求，验结果，再迭代 |
| 失败原因 | 代码不会写 | 需求不清、边界不清、验收不清 |

## 它不等于什么

Vibe Coding 不等于：

- 只说一句“帮我做个 App”就能稳定得到成品
- 不需要测试
- 不需要理解代码
- 不需要产品判断和取舍

真正有效的 Vibe Coding，靠的是：

- 目标清晰
- 约束清晰
- 分步推进
- 每一步都验收

## 新手最容易误解的地方

### 误解 1：AI 会自动理解我脑子里的需求

不会。AI 只能根据你说出来的内容工作。

你说得越模糊，结果越随机。

### 误解 2：AI 第一次写出来就应该是最终版

不现实。第一版更像草稿或方向稿。

真正的开发过程通常是：

1. 先让 AI 产出方案
2. 再让 AI 做最小实现
3. 再根据结果修
4. 最后补测试、边界和文档

### 误解 3：会 Vibe Coding 就不需要懂一点技术

也不对。你至少要懂：

- 文件和目录是什么
- Git 和 GitHub 基本怎么用
- 看错误信息
- 知道什么叫“验收通过”

## 一个更准确的定义

Vibe Coding 本质上是：

**用自然语言驱动开发流程。**

你在里面扮演的不是“打字员”，而更像：

- 产品经理：说清需求
- 架构师：判断方案
- 测试工程师：发现问题
- 项目经理：控制节奏

## 一套能落地的工作流

### 第 1 步：先说目标，不要先说实现

❌ 不推荐：

> 帮我用 React、Node、Postgres、Tailwind、Redis 做一个网站

这类说法把技术名词堆上来了，但目标不清楚。

✅ 更推荐：

> 做一个个人知识库网站。用户可以新增、编辑、搜索文章。先做一个本地可运行的 MVP，不考虑登录。

先说目标，再谈技术，AI 才知道要解决什么问题。

### 第 2 步：把边界讲清楚

至少讲清这些：

- 做给谁用
- 先做什么，不做什么
- 运行环境是什么
- 结果长什么样算“可以”

例子：

> 这是给我自己用的桌面小工具。先只支持本地 Markdown 文件，不接数据库。能搜索标题和正文就算第一版完成。

### 第 3 步：让 AI 先出方案，不要一上来就写全量代码

可以直接说：

> 先不要写代码。先给我一个最小实现方案，列出目录结构、关键模块和分步骤计划。

这样你能先看方向对不对，再决定是否动手。

### 第 4 步：要求它分阶段实现

推荐的推进顺序：

1. 先搭最小骨架
2. 再补核心功能
3. 再补异常处理
4. 再补测试和文档

不要一次让 AI 做“全套产品 + 完美体验 + 上线部署”。

### 第 5 步：每一轮都做验收

每轮结束后至少检查：

- 能不能运行
- 结果是不是符合需求
- 有没有明显 bug
- 文件结构有没有乱
- 文档有没有同步

### 第 6 步：用“基于结果的反馈”继续迭代

❌ 不推荐：

> 不太对，你再改改

✅ 推荐：

> 现在首页可以打开，但有 3 个问题：1. 搜索框没有生效。2. 移动端布局溢出了。3. 没有空状态提示。请只修这 3 个问题，不要重构其他部分。

## 提需求时的推荐模板

你可以直接套这个格式：

```text
目标：
我要做一个什么东西？

用户：
这是给谁用的？

第一版范围：
必须做什么？
暂时不做什么？

技术约束：
必须用什么？
不能用什么？

验收标准：
什么结果算完成？
```

## 一个完整示例

### 模糊版本

> 帮我做一个博客

问题：

- 不知道给谁用
- 不知道需要哪些页面
- 不知道是否需要后台
- 不知道是否移动端适配

### 可执行版本

> 帮我做一个个人博客 MVP。  
> 目标用户是访客和我自己。  
> 第一版只需要首页、文章列表页、文章详情页。  
> 不需要后台管理，不需要评论系统。  
> 技术上优先简单，能本地运行即可。  
> 验收标准是：我能启动项目、看到 3 个页面、点击文章能进入详情页、手机宽度下不破版。

这时候 AI 才比较容易给出靠谱结果。

## 什么时候适合 Vibe Coding

- 做原型
- 做个人工具
- 做内部小系统
- 做 Demo
- 学习新框架
- 重复性改动很多的任务

## 什么时候要谨慎

- 金融、医疗、法律等高风险系统
- 安全要求很高的功能
- 非常复杂的底层性能优化
- 多人协作且规则还没定清楚的大项目

这些场景不是不能用 AI，而是不能只靠“感觉推进”。

## 常见失败模式

### 1. 一次提太大

“帮我做一个完整 SaaS 平台”这种请求，通常只会把项目做散。

### 2. 没有验收标准

你不说“什么算完成”，AI 就会自己猜。

### 3. 不看生成结果

AI 写出来就直接用，后面返工会更大。

### 4. 不做版本管理

没有 Git，你很难回退和比较改动。

### 5. 不同步文档

越到后面越说不清“为什么现在长这样”。

## 给新手的最低配建议

如果你现在刚开始：

1. 先学会 Git 基础
2. 学会 GitHub 仓库和 PR
3. 学会复制模板、运行脚本、看报错
4. 再开始做 Vibe Coding

## 一句话总结

Vibe Coding 不是“不会开发也能直接做成一切”。

它更像是：

**把开发从“手写实现”升级为“用自然语言指挥、验证和迭代”。**
