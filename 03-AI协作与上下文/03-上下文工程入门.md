# 上下文工程入门

这篇文档讲的是：

**在 AI Coding 里，怎样让 AI 持续拿到正确上下文，而不是每次都从零猜。**

如果说 Git 解决的是“版本怎么管”，那么上下文工程解决的是：

**AI 到底依据什么做判断。**

---

## 1. 什么是上下文工程

一句话理解：

> 把 AI 做事需要的关键信息，稳定、分层、可维护地组织起来。

很多人以为 AI Coding 的核心是“提示词写得会不会花哨”，其实更关键的是：

- 项目规则有没有写出来
- 目录结构是不是清楚
- 任务状态有没有沉淀
- 历史决策能不能找到
- AI 当前是否拿到了正确文件

这些合起来，才是上下文工程。

---

## 2. 为什么 Vibe Coding 很依赖上下文工程

Vibe Coding 的问题不在于“AI 不会写”，而常常在于：

- AI 不知道项目规则
- AI 不知道哪些文件重要
- AI 不知道这次任务的边界
- AI 不知道之前为什么这么设计

所以更准确地说：

**Vibe Coding 的稳定性，取决于上下文工程的质量。**

没有上下文工程，AI 很容易：

- 每轮都重复问同样的问题
- 输出风格漂移
- 改坏原来的约定
- 把临时想法当成长期规则

---

## 3. 在这个项目里，上下文主要来自哪里

你可以把上下文来源理解成 7 层：

| 层级 | 文件或来源 | 作用 |
|------|------------|------|
| 1 | 当前对话 | 当前任务和临时要求 |
| 2 | 仓库结构 | AI 理解项目边界和模块分工 |
| 3 | `CLAUDE.md` | Claude Code 的项目级记忆 |
| 4 | `AGENTS.md` | 团队协作规则和文档协议 |
| 5 | `SKILL.md` | 某类任务的专用流程 |
| 6 | `TASKS.md` / `DECISIONS.md` / `CHANGELOG.md` | 当前状态、决策原因、变更历史 |
| 7 | `references/`、官方文档、外部资料 | 按需加载的补充知识 |

这也是为什么一个成熟项目不能只靠聊天记录。

---

## 4. `AGENTS.md`、`CLAUDE.md`、Rules 之间是什么关系

可以这样记：

- `CLAUDE.md`：Claude Code 的项目记忆入口
- `AGENTS.md`：团队写给代理的协作章程
- Rules：这些文件里真正约束行为的规则内容

也就是说：

- `AGENTS.md` 和 `CLAUDE.md` 是“载体”
- Rules 是“内容”

例如这几句都算 Rules：

- 提交前先看 `git diff`
- 只用中文写文档
- 不要提交 `.env`
- 任务完成后必须更新 `TASKS.md`

---

## 5. 什么信息应该写进文件，而不是只留在聊天里

更适合写进文件的内容：

- 长期有效的项目规则
- 目录结构和命名约定
- 提交流程
- 常用命令
- 当前任务状态
- 重要决策原因
- 可复用模板

只适合留在聊天里的内容：

- 这一次临时的小修改
- 还没确认的试探想法
- 很快就会过期的中间讨论

一个简单判断标准：

> 下次开新会话还需要它吗？

如果答案是“需要”，大概率就该落文件。

---

## 6. 新手最实用的上下文工程工作流

### 第一步：先定义项目级规则

至少明确：

- 文档语言
- Git 工作流
- 命名规范
- 敏感信息规则
- 任务结束要更新哪些文件

### 第二步：把规则放进固定文件

推荐分工：

- `CLAUDE.md`：精简项目记忆
- `AGENTS.md`：完整协作协议
- `SKILL.md`：某个 Skill 的专用规则

### 第三步：把“变化中的上下文”单独管理

不要把这些混进 `CLAUDE.md`：

- 当前任务
- 今日进展
- 临时 blocker

这些更适合写进：

- `TASKS.md`
- `DECISIONS.md`
- `CHANGELOG.md`

### 第四步：每次任务结束都更新

上下文工程不是“项目启动时写一次”，而是每轮都维护。

---

## 7. 一个最小可用结构

对一个 AI Coding 项目，最低配可以是：

```text
project/
├── CLAUDE.md
├── AGENTS.md
├── CHANGELOG.md
├── TASKS.md
└── DECISIONS.md
```

如果是 Skill 项目，再加：

```text
skills/my-skill/
├── SKILL.md
├── CHANGELOG.md
├── ROADMAP.md
├── TASKS.md
├── DECISIONS.md
├── scripts/
├── references/
└── assets/
```

---

## 8. 上下文工程里最常见的错误

### 错误 1：把所有规则都堆进 `CLAUDE.md`

结果就是又长又乱，真正重要的规则反而不明显。

### 错误 2：没有区分长期规则和当前任务

长期规则应稳定，当前任务应独立更新。

### 错误 3：文件很多，但没有明确职责

不是文件越多越好，而是每个文件回答的问题要明确。

### 错误 4：只新增文档，不维护文档

过期文档比没有文档更危险。

---

## 9. 对 Vibe Coding 最有帮助的几个动作

1. 把项目规则写进 `AGENTS.md` 或 `CLAUDE.md`
2. 把当前任务写进 `TASKS.md`
3. 把关键决策写进 `DECISIONS.md`
4. 把专用流程写成 `SKILL.md`
5. 每轮结束后同步文件，而不是只在聊天里说

## 一句话总结

上下文工程的本质不是“多写几个规则文件”，而是让 AI 在每一轮都能拿到正确、稳定、可追溯的项目信息。
