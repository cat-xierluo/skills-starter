# 许可证与开源授权参考

当你需要确认“没写许可证意味着什么”“应该用哪个许可证”“`license` 字段该怎么写”时，优先看这些官方或准官方入口。

## 推荐先看

### 1. 选许可证

- [Choose a License](https://choosealicense.com/)

适合快速判断 `MIT`、`Apache-2.0`、`GPL-3.0` 这类常见许可证的差异。

### 2. 理解“没写许可证”的后果

- [No License](https://choosealicense.com/no-permission/)

适合理解为什么“仓库公开”不等于“默认允许复制和商用”。

### 3. 确认标准许可证标识

- [SPDX License List](https://spdx.org/licenses/)

适合确认 `SKILL.md` 里 `license` 字段应该使用什么标准写法。

### 4. 在 GitHub 仓库里添加许可证

- [Licensing a repository](https://docs.github.com/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/licensing-a-repository)

适合确认 GitHub 仓库里如何补 `LICENSE` 文件。

### 5. 查看被广泛认可的开源许可证

- [Open Source Initiative Licenses](https://opensource.org/licenses)

适合进一步确认某个许可证是否属于 OSI 认可的开源许可证。

## 在 Skill 项目里怎么用这些资料？

- 写 `SKILL.md` 时，用 SPDX 标识写 `license`
- 新增 `LICENSE.txt` 时，使用对应许可证的正式文本
- 评估第三方 Skill 时，先看仓库有没有明确许可证
- 不确定兼容性时，不要直接复制第三方实现进自己的仓库

## 选择时优先确认的维度

- 是否允许商业使用
- 是否要求保留署名
- 是否要求衍生作品继续按相同方式开放
- 是否需要更明确的专利授权表达
- 是否希望未来保留单独商业授权或双重授权空间

## 常见许可证快速对照

| 许可证 | 商用 | 署名/保留许可 | 派生版本是否要继续开放 | 备注 |
|------|------|----------------|------------------------|------|
| `MIT` | 允许 | 需要 | 不要求 | 最常见宽松代码许可证 |
| `Apache-2.0` | 允许 | 需要 | 不要求 | 比 MIT 更强调专利授权表述 |
| `GPL-3.0` | 允许 | 需要 | 通常要求 | 分发修改版时约束较强 |
| `AGPL-3.0` | 允许 | 需要 | 通常要求 | 网络服务场景也要重点看义务 |
| `CC-BY` | 允许 | 需要 | 不要求 | 常见于文档、课程、内容 |
| `CC-BY-SA` | 允许 | 需要 | 要求 | 适合希望改编后也继续开放的内容 |
| `CC-BY-NC-SA-4.0` | 不允许直接商用 | 需要 | 要求 | 适合想保留非商用边界和后续授权空间 |
| `CC-BY-ND` / `CC-BY-NC-ND` | 视具体条款 | 需要 | 不允许派生分发 | 对需要二次修改的 Skill 资料通常不友好 |

## 派生相关条款要特别看什么？

- `SA` / `ShareAlike`：改编后继续按相同方式共享
- `GPL` / `AGPL`：分发修改版时，通常要继续开放对应代码
- `AGPL`：如果修改版通过网络提供服务，也要特别关注开放义务
- `ND` / `NoDerivatives`：不允许公开分发改编版本
- `NC` / `NonCommercial`：不允许直接商业使用，但“什么算商业使用”要谨慎理解

## 参照 `legal-skills` 值得抽取的实践

- 在多 Skill 清单里显式展示许可证，而不是只写名称和说明
- 在 `AGENTS.md` 中把许可证当成维护规范，而不是临发布前才补
- 对官方或迁移 Skill，保持上游许可证，不做擅自改写
- 许可证决策应围绕商用边界、署名义务和授权策略，而不是只按 Skill 类型机械套用
