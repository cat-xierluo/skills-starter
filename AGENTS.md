# AGENTS.md - 项目规范

本项目不是单纯的模板仓库，而是一个面向 AI 与人类协作者的 Skill 创作 starter、需求分流入口和上下文仓库。
AI 代理应把它当作“先判断是否已有现成方案，再决定复用、改造还是从零创建”的执行环境。

---

## 核心原则

- **中文优先**：回复使用中文，文档使用中文。
- **文档即上下文**：关键信息写在文件里，不靠记忆。
- **先搜索，后创造**：遇到问题先查现有文档，再创新方案。
- **先复用，后定制**：能直接使用现成 Skill，就不要重复造轮子；只有在现有方案不匹配时才创建新 Skill。

---

## README 与 AGENTS 的分工

- `README.md`：给人看，说明仓库能做什么、默认怎么使用。
- `AGENTS.md`：给 AI 看，把 README 中的人类流程转成可执行的检索与决策协议。
- `CLAUDE.md`：项目入口配置，可包含或转发 `AGENTS.md` 规则，但不替代本文件。

---

## 上游同步文件

- `04-创建Skill/SKILL-*-GUIDE.md` 视为从 `legal-skills` 同步的上游指南文件。
- 除非用户明确要求“同步上游内容”或“直接改这些 guide 文件”，否则不要手动修改这些文件。
- 如果当前任务只是补充解释、导航或示例，优先更新 `README.md`、`AGENTS.md`、概念文档、工具文档或参考资料，不要改上游同步文件。

---

## AI 默认工作流

处理需求时，默认按下面顺序执行：

1. **先判断需求类型**
   - `找现成 Skill`
   - `在现有 Skill 基础上调整`
   - `从零创建 Skill`
   - `补概念、工具或上下文知识`
2. **先搜索现成方案**
   - 优先使用项目内置 `.claude/skills/find-skill/`
   - 本仓库采用 `.claude/skills -> ../skills` 的符号链接结构，根目录 `skills/` 是单一来源
   - 仍不够时，再使用 Skills CLI，例如 `npx skills find <keyword>`
3. **能复用就不要新建**
   - 如果已有 Skill 基本满足需求，优先直接使用
   - 如果只有局部不匹配，优先做定向改造
4. **确实需要新建时，再进入创建流程**
   - 优先查阅 `04-创建Skill/`
   - 优先复用 `skills/skill-template/`
   - 需要完整规范时查阅 `04-创建Skill/SKILL-DEV-GUIDE.md`
   - 涉及多 Skill 协作时查阅 `04-创建Skill/SKILL-ORCHESTRATION-GUIDE.md`

## 仓库检索协议

在仓库内查找上下文时，不要无差别全量扫描，先按问题类型选择目录：

| 目录 | 适用问题 |
| :--- | :--- |
| `01-概念入门/` | Skill 是什么、术语是什么、基础心智模型是什么 |
| `02-工具指南/` | Git、GitHub、SSH、commit 规范、推送流程 |
| `03-AI协作与上下文/` | Vibe Coding、`AGENTS.md`、`CLAUDE.md`、Rules、上下文工程 |
| `04-创建Skill/` | 需求分析、搜索现成方案、复制模板、调试发布 |
| `05-参考资料/` | Claude 官方文档、博客、PDF、最佳实践、深度资料 |

默认检索顺序：

1. 先看 `README.md` 中的资源导航，确认应该去哪一组目录。
2. 再在对应目录内按文件名和关键词检索。
3. 涉及官方依据或深度背景时，再进入 `05-参考资料/`。
4. 只有目录边界不清时，才做跨目录扩展检索。

检索时优先：

- 看文件名是否已经能回答问题
- 在目标目录内用关键词检索
- 命中后只打开相关文件，不批量读取无关材料

## Project-local Skills 约定

- 本仓库采用 `legal-skills` 同款结构：`.claude/skills` 是指向 `../skills` 的符号链接。
- 根目录 `skills/` 是 Skill 源文件的单一来源，Claude Code 通过 `.claude/skills/` 读取这些 Skill。
- `skills/` 下既可以放可直接调用的 Skill，也可以放模板和示例；是否调用，取决于 `SKILL.md` 的描述与当前任务匹配程度。
- `.claude/settings.local.json`、`.claude/*.local.*`、`.claude/.git/` 等本地配置和元数据继续保持忽略。

## 许可证管理规范

参照 `legal-skills` 的维护经验，许可证不应被当成“最后再补的装饰”，而应在创建或迁移 Skill 时尽早确定。

### 选择许可证时优先看这几个维度

不要只按 Skill 类型机械套许可证，先判断你的授权目标：

1. **是否允许商业使用**
   - 如果你一开始就允许任何人商用，通常会更偏向 `MIT` 或 `Apache-2.0`
   - 如果你想保留后续单独商业授权、双重授权或收费授权空间，就不要轻易给出无限制商用许可
2. **是否要求署名**
   - 大多数开源或开放内容许可证都要求保留版权和许可声明
   - 如果你非常在意署名展示方式，应在 `LICENSE.txt` 或 README 中写清楚
3. **是否要求衍生作品继续按相同方式开放**
   - 如果你希望修改版继续保持同类开放方式，可考虑带 share-alike / copyleft 的许可证
   - 如果不要求，通常会更偏向宽松许可证
4. **是否涉及专利授权或企业合规表达**
   - 需要更明确专利授权表述时，通常优先考虑 `Apache-2.0`
5. **是否是官方 Skill 或第三方迁移 Skill**
   - 保持上游许可证，不要擅自改写为自己的许可证
   - 需要时可在 frontmatter 中写 `license: Complete terms in LICENSE.txt`

### 常见选择方向

- **`MIT`**：宽松，允许商用，要求保留版权和许可声明，适合希望最大化传播和复用
- **`Apache-2.0`**：同样宽松，但专利授权表述更完整，适合团队或公司项目
- **`GPL-3.0`**：如果分发修改版，通常也要继续开源，适合强调 copyleft
- **`CC-BY` / `CC-BY-NC-SA-4.0`**：更常用于文档、知识内容、方法论材料；如果你想保留非商用边界或后续商业授权空间，可以重点评估
- **私有 / 自定义条款**：如果暂时不打算开放复用，或需要更细的商业限制，可以保持私有或单独写授权条款

### `license` 字段与 `LICENSE.txt`

- 对外共享、分发或长期维护的 Skill，默认应包含 `LICENSE.txt`
- `SKILL.md` 中的 `license` 字段应与 `LICENSE.txt` 保持一致
- 如果采用限制商用或其他附加条件的许可证，应在 `LICENSE.txt` 或 README 中写清适用范围和联系渠道
- 如果你未来希望通过商业授权获利，应提前考虑：当前许可证是否已经把商用权直接放出去了
- 如果许可证尚未确定，允许先标注“待确认”，但不要假装已经完成授权设计

### README 中的许可证信息

- 如果仓库维护多个 Skill 的列表、目录表格或索引表，建议显式加入“许可证”列
- 目标不是做法律展示，而是让人和 AI 在复用前就知道能否商用、能否改造、是否要保留原许可

## 目录命名规范

- **目录名**：用中文（如 `概念入门`、`工具指南`、`AI协作与上下文`、`创建Skill`、`参考资料`）
- **文件名**：用中文（如 `01-什么是-Skill.md`、`02-目录结构.md`）
- **代码文件除外**：`SKILL.md`、`AGENTS.md`、`CLAUDE.md` 等保持原样

## 文档编写规范

### SKILL.md

Skill 定义文件，放在技能根目录。

```markdown
---
name: skill-name        # 英文，唯一标识
description: 英文描述   # 英文，说明用途
---

# Skill Name

（正文用中文编写）
```

### 其他文档

- 语言：中文
- 代码块：使用 \`\`\`
- 列表：用 `-` 或 `1.` 均可

## 协作文档

本仓库和复杂 Skill 项目默认使用以下协作文档：

- `README.md`：记录人类读者的默认使用路径和资源导航
- `CHANGELOG.md`：记录对外可见的变更，并按 `0.1.0`、`0.2.0` 这类内部迭代版本持续维护，即使尚未正式发布
- `docs/ROADMAP.md`：记录阶段目标和里程碑
- `docs/ARCHITECTURE.md`：记录结构、职责和边界
- `status/TASKS.md`：记录当前待办和完成状态
- `status/DECISIONS.md`：记录决策与工作日志

工作涉及结构性调整、模板更新、默认工作流变化或可见行为变化时，应同步更新相关文档。

## Git 工作流

### 分支命名

```
refactor/xxx   # 重构
feature/xxx   # 新功能
bugfix/xxx     # 修复
```

### Commit 规范

- 语言：中文
- 风格：简洁，说明做了什么

```
feat: 新增 xxx 功能
fix: 修复 xxx 问题
refactor: 重构 xxx
docs: 更新 xxx 文档
```

### PR 流程

1. 创建 PR
2. 常规功能：创建后自动合并
3. 重大变更：人工 review 后合并
