# 什么是许可证？

许可证（License）是作者提前写清楚的使用规则。

它回答的是：

- 别人能不能复制你的 Skill
- 能不能修改
- 能不能商用
- 能不能二次分发
- 修改后要不要继续开源

## 为什么 Skill 仓库必须关心许可证？

Skill 仓库往往不只有代码，还会包含：

- `SKILL.md`
- `references/` 里的参考资料
- `scripts/` 里的脚本
- `assets/` 里的模板和静态资源
- 文档、截图、示例输出

这些内容是否允许别人拿去复用，不会因为仓库公开就自动变清楚。  
**公开可见** 和 **允许复用** 不是一回事。

## 最容易被误解的一点

**没有写许可证，不等于“默认大家都能用”。**

更接近的理解是：作者并没有主动授予复制、修改和再分发的权利。  
所以看到一个公开仓库，不代表你就可以直接把里面的内容复制进自己的 Skill。

## 常见许可证怎么选？

### MIT

- 很宽松
- 允许商用、修改、闭源再分发
- 需要保留原始版权和许可声明
- 适合：想让更多人低门槛复用

### Apache-2.0

- 也比较宽松
- 除了常规复用外，对专利授权的表述更清楚
- 适合：团队或公司项目，比较在意专利风险表达

### GPL-3.0

- 更强的 copyleft
- 分发修改版时，通常也要继续按 GPL 公开
- 适合：希望派生版本继续保持开源

### 不开放或仅内部使用

- 可以不选开源许可证
- 但最好明确写内部使用条款，或保持私有仓库
- 适合：公司内部流程、敏感知识库、暂不打算对外复用的 Skill

## 常见许可证对比表

下面这张表不是法律意见，而是做 Skill 时最常用的工程判断表。

| 许可证 | 更常用于 | 是否允许商用 | 是否要求署名/保留许可 | 修改后是否必须同协议开放 | 网络服务是否也触发开放要求 | 对后续商业授权空间的影响 |
|------|----------|--------------|------------------------|--------------------------|----------------------------|--------------------------|
| `MIT` | 代码、工具 | 是 | 是 | 否 | 否 | 较小，因商用权已广泛放出 |
| `Apache-2.0` | 代码、团队项目 | 是 | 是 | 否 | 否 | 较小，但专利表述更完整 |
| `GPL-3.0` | 代码 | 是 | 是 | 是 | 通常不因纯网络使用触发 | 中等，分发修改版时约束较强 |
| `AGPL-3.0` | 服务端代码、网络软件 | 是 | 是 | 是 | 是 | 中等偏强，网络服务场景约束更强 |
| `CC-BY` | 文档、教程、知识内容 | 是 | 是 | 否 | 不适用代码语境 | 较小，商用与改编都放得较开 |
| `CC-BY-SA` | 文档、知识内容 | 是 | 是 | 是 | 不适用代码语境 | 中等，改编后要继续同协议 |
| `CC-BY-NC-SA-4.0` | 文档、方法论、内容资产 | 否 | 是 | 是 | 不适用代码语境 | 较大，更适合保留后续商业授权空间 |
| 私有 / 自定义条款 | 内部项目、待商业化项目 | 由你决定 | 由你决定 | 由你决定 | 由你决定 | 最大，但复用门槛也最高 |
| 未声明许可证 | 任何内容 | 不应默认视为允许 | 不明确 | 不明确 | 不明确 | 风险最高，不建议直接复用 |

## 派生版本 / 衍生作品到底是什么意思？

很多人最容易忽略的，不是“能不能看”，而是“别人改了以后还需要遵守什么”。

### 宽松许可证

以 `MIT`、`Apache-2.0` 为代表：

- 允许别人修改、再分发、商用
- 通常只要求保留版权和许可声明
- 修改后的版本不必继续使用同一个许可证

### Copyleft / Share-Alike

以 `GPL-3.0`、`AGPL-3.0`、`CC-BY-SA`、`CC-BY-NC-SA-4.0` 为代表：

- 允许修改和再利用
- 但如果你把修改版再分发，通常不能随便换成更封闭的协议
- 这类许可证的核心是：你可以改，但改出来再分享时，要继续按相近规则开放

### No Derivatives

以 `CC-BY-ND`、`CC-BY-NC-ND` 为代表：

- 允许转载原作品
- 但不允许公开分发改编版本
- 对需要二次定制、改写、拼装的 Skill 或文档来说，通常不够友好

## 如果你以后想商业化，要先想什么？

- 如果你已经用了 `MIT` 或 `Apache-2.0`，别人通常可以直接商用，你后面再卖“商用授权”的空间会比较有限
- 如果你想保留企业授权、双重授权、非商用转商用收费这类路径，就要在一开始考虑更谨慎的授权方式
- 如果你的 Skill 同时包含代码、文档、案例、模板，可能需要分别考虑授权，不一定所有内容都用同一许可证

## 不要只按 Skill 类型选许可证

比起先问“这是工具型 Skill 还是内容型 Skill”，更重要的是先问：

### 1. 你要不要允许别人直接商用？

- 如果答案是“可以”，那通常会更接近 `MIT` 或 `Apache-2.0`
- 如果答案是“我还想保留后续收费授权或商业合作空间”，那就不能随便给出无限制商用许可

### 2. 你要不要别人保留署名？

- 如果你很在意署名、来源和引用方式，就要看许可证是否要求保留版权和许可声明
- 还要在 README 或 `LICENSE.txt` 里把署名要求写清楚

### 3. 你要不要别人修改后也继续开放？

- 如果你希望派生版本继续按相同方式开放，就要考虑 share-alike 或 copyleft
- 如果你不要求这一点，就更可能偏向宽松许可证

### 4. 你以后会不会想做商业化或授权收费？

- 这是很多人一开始会忽略的点
- 如果你已经给出了允许任何人商用的宽松许可证，后面再想靠“商用授权”收费，空间就会小很多
- 如果你本来就想保留单独授权、企业授权或双重授权的可能性，应该一开始就按这个目标来设计

### 5. 这个 Skill 是你原创的，还是迁移来的？

- 官方 Skill 或第三方 Skill，要优先保持上游许可证
- 不要因为自己整理、翻译、改写了一遍，就默认能改成自己的许可证

## 一个更实用的判断顺序

1. 先想清楚：要不要开放商用
2. 再想清楚：要不要强制署名和同协议共享
3. 再看：后面是否要留商业授权空间
4. 最后才是选具体许可证名字

## 在 Skill 项目里具体怎么落地？

最少做到这 4 件事：

1. 在仓库根目录新增 `LICENSE.txt` 或 `LICENSE`
2. 在 `SKILL.md` frontmatter 中按需补 `license`
3. 在 README 或文档里说明是否允许商用、修改和再分发
4. 检查依赖、素材、示例数据和外部引用是否与自己的许可证兼容

如果你要在 `SKILL.md` 里写 `license`，推荐使用 SPDX 常见标识，例如：

- `MIT`
- `Apache-2.0`
- `GPL-3.0-only`

## Skill 开发里最容易漏掉的风险

- 代码是你写的，但图标、截图、图片素材不是你授权的
- 依赖库能用，但它的许可证和你的分发方式不兼容
- 你把别人的 prompt、规则文档、案例文本直接复制进仓库
- 你以为“只要在 GitHub 上公开就算开源”

## 实用建议

- 想尽量降低复用门槛，优先考虑 `MIT` 或 `Apache-2.0`
- 想要求分发后的修改版继续开源，再考虑 `GPL-3.0`
- 如果你想保留后续商业授权空间，不要轻易给出无限制商用许可
- 许可证不明确的第三方 Skill，不要直接复制实现
- 如果代码和文档打算分开授权，要在 README 中明确写清楚
- 不确定时先标记“待确认”，不要随手填一个不准确的许可证
- 如果仓库维护多个 Skill 列表，最好在表格里直接加“许可证”列
