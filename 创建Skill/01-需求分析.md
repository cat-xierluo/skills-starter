# 01 需求分析

在动手写代码之前，先理解真正要解决什么问题。

## 如何理解用户需求

1. **明确场景**：这个 Skill 是给谁用的？在什么情况下会被触发？
2. **拆解动作**：用户想完成什么具体操作？输入是什么，输出是什么？
3. **边界确认**：什么情况不归这个 Skill 管？

> 不要假设需求，提前和用户确认。

## 评估需求复杂度

| 复杂度 | 特征 | 处理方式 |
|--------|------|----------|
| 低 | 单工具调用，逻辑简单 | 直接实现 |
| 中 | 2-3 个工具组合，需要判断分支 | 先写 SKILL.md，再实现 |
| 高 | 多步骤、多状态、外部依赖 | 拆分成多个 Skill 或子任务 |

**如果一个 Skill 需要超过 200 行代码才能说清楚，说明它太大了。**

## 判断是否需要创建新 Skill

满足以下任一条件，考虑创建独立 Skill：
- 独立使用场景，不依赖其他 Skill
- 需要单独的配置或状态
- 可能被多个调用方复用
- 团队中有多人需要

满足以下条件，考虑合并到现有 Skill：
- 只是现有 Skill 的小功能扩展
- 两个功能强绑定，拆分后反而更复杂
- 使用频率极低

## 需求文档怎么写

不需要模板，但必须包含：

```
## 目标
[一句话说清楚这个 Skill 要做什么]

## 触发场景
[什么情况下用户/AI 会调用这个 Skill]

## 输入
[需要什么参数，格式是什么]

## 输出
[返回什么结果]

## 注意事项
[边界情况、坑、已知限制]
```

好的需求文档让你在写代码时不需要再问任何人。
